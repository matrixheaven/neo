# Neo Multi-Agent Design

## Overview

Neo should implement Multi-Agent as a first-class runtime feature with a Codex-like
internal model and a Kimi-like default user experience.

The internal model is a persistent agent tree: subagents have stable IDs, display
names, parent/child relationships, state, mailboxes, cancellation, and resumable
results. The user experience is transcript-native: delegation appears as normal
tool output in the main chat, with live child-agent progress rendered inline.

The default behavior is foreground and blocking. When the main agent delegates
work, it waits for the delegated agent or swarm unless the tool call explicitly
requests background execution. Background mode still exists for Codex-style
parallel collaboration, but it is opt-in.

This design intentionally combines:

- Codex's durable agent tree and collaboration controls.
- Kimi Code's blocking delegation, swarm scheduler, and transcript UX.
- OpenCode's agent profile and permission ideas.
- Claude Code's dynamic workflow concept as a later harness layer, implemented
  in a Rust-friendly scripting model.

## Goals

- Add a product-grade Multi-Agent runtime to Neo, not a debug console.
- Default `Delegate` and `DelegateSwarm` to foreground blocking behavior.
- Support explicit background agents that can be listed, waited on, interrupted,
  and messaged later.
- Render delegate and swarm state as inline transcript tool cards.
- Keep swarm progress visible while the main agent is blocked.
- Preserve Bayesian-style progress estimation for partially completed swarms.
- Use stable internal IDs plus deterministic hardcoded display names.
- Provide an incremental V1-to-final roadmap that avoids rewrites.
- Add a later workflow/harness layer using one canonical scripting language.
- Keep subagents inside Neo's existing permission, approval, session, and
  workspace safety model.

## Non-Goals

- No separate swarm page, side panel, or hidden alternate transcript.
- No LLM-generated subagent names.
- No compatibility branch for old experimental names once the canonical tool
  names land.
- No model-facing raw RPC/debug command such as `agent_call` or `inspect_agent`.
- No hosted coordination service, remote worker marketplace, profile sync, or
  cloud collaboration.
- No YAML/Lua/JS multi-language workflow compatibility layer.
- No autonomous git mutation by subagents. Subagents must not run git mutations
  such as commit, reset, checkout, stash, clean, rebase, merge, push, or branch
  deletion.

## 1. Product Model

Neo has one Multi-Agent runtime with two main tool surfaces:

| Tool | Default | Purpose |
| --- | --- | --- |
| `Delegate` | foreground | Send one bounded task to one subagent. |
| `DelegateSwarm` | foreground | Send many related bounded tasks to a coordinated swarm. |

Both tools can opt into background execution when the parent needs parallel
collaboration instead of blocking aggregation.

```text
Neo Main Agent
  |
  | calls Delegate / DelegateSwarm
  v
MultiAgent Runtime                 TUI Transcript
  |                                |
  |-- Agent Registry               |-- inline tool card
  |-- Foreground Join              |-- live child transcript
  |-- Mailbox / Followup           |-- swarm progress rows
  |-- Wait / Interrupt / List      |-- Bayesian progress estimate
  |-- Background Residency         |
  v
Subagents: Zeno, Gibbs, Hokke, Laber, Atlas, Wukong...
```

Foreground is the default because it protects the mainline agent from racing
ahead and editing the same code before delegated work finishes. Background is
still important, but it must be explicit.

## 2. Runtime Architecture

### `MultiAgentRuntime`

Owned by the interactive/session runtime. It is responsible for:

- allocating stable agent IDs
- assigning deterministic display names
- creating child agent sessions
- tracking parent/child relationships
- enforcing foreground joins
- managing background residency
- delivering mailbox messages
- interrupting child turns
- aggregating child results
- emitting transcript events for TUI rendering

### Agent Identity

Each subagent has a stable machine identity and a deterministic display identity.

```text
AgentIdentity
  id: AgentId                 # stable internal ID, never generated by the LLM
  display_name: String        # from built-in pool, e.g. Gibbs
  path: AgentPath             # /root/Gibbs or /root/swarm-1/Gibbs
  parent_id: AgentId | root
  role: AgentRole             # coder, explorer, planner, harness, reviewer
```

Internal IDs drive storage, resume, mailbox routing, and event correlation.
Display names are UI sugar only.

### Display Name Pool

Neo ships a large hardcoded English name pool. The LLM never generates names.
This protects prompt/cache stability and keeps the UI pleasant.

Initial pool examples:

```text
Zeno, Gibbs, Hokke, Laber, Ada, Turing, Knuth, Shannon,
Euler, Noether, Gauss, Hypatia, Athena, Hermes, Apollo,
Atlas, Merlin, Arthur, Wukong, Nezha, Mulan, Orion,
Kepler, Curie, Feynman, Lovelace, Hopper, Ramanujan,
Socrates, Plato, Artemis, Diana, Minerva, Loki, Freya
```

Rules:

- A session assigns names deterministically from the pool.
- A swarm never repeats a display name within the same swarm.
- A resumed agent keeps its prior display name.
- If the pool is exhausted, append a numeric suffix deterministically.
- Names are not part of protocol compatibility; only IDs are canonical.

## 3. Tool Semantics

### `Delegate`

`Delegate` sends one bounded task to one subagent.

```text
Delegate
  task: string
  role?: coder | explorer | planner | reviewer | harness
  mode?: foreground | background   # default foreground
  context?: inherit | summary | none
```

Foreground behavior:

1. Parent creates the subagent.
2. Parent blocks on the child result.
3. TUI renders the child transcript inside the parent tool card.
4. Parent receives the child summary/result when the child completes.
5. While the child is running, the user can detach it with `Ctrl+B`.

Background behavior:

1. Parent creates the subagent.
2. Tool returns immediately with agent ID and display name.
3. Parent can continue its own turn.
4. Completion enters the mailbox and TUI task state.
5. Parent can later call wait/list/followup/interrupt controls.
6. The background agent appears in the `/tasks` Task Browser.

Detach behavior:

1. A foreground `Delegate` starts in blocking mode.
2. If the user presses `Ctrl+B`, Neo converts the same child run into a
   background delegate.
3. The parent join is released, so the main conversation can continue.
4. The transcript card remains in place and changes to background-running state.
5. The child is registered in the shared background task surface.
6. Detach does not restart the child or create a second agent record.

### `DelegateSwarm`

`DelegateSwarm` sends related bounded tasks to many subagents.

```text
DelegateSwarm
  description: string
  items: string[]
  prompt_template: string           # contains {{item}}
  role?: coder | explorer | planner | reviewer
  mode?: foreground | background    # default foreground
  max_concurrency?: number
```

Foreground behavior:

1. Parent creates a swarm run.
2. Subagents start under a concurrency cap.
3. Parent blocks until the swarm reaches terminal state.
4. TUI renders live progress for every child.
5. Tool returns an aggregate result with per-agent outcomes.
6. While the swarm is running, the user can detach it with `Ctrl+B`.

Background behavior:

1. Parent creates a swarm run.
2. Tool returns the swarm ID immediately.
3. Swarm continues under the scheduler.
4. Parent can list/wait/interrupt/follow up.
5. Completion enters the mailbox.
6. The background swarm appears in the `/tasks` Task Browser.

Detach behavior:

1. A foreground `DelegateSwarm` starts in blocking mode.
2. If the user presses `Ctrl+B`, Neo converts the same swarm run into a
   background swarm.
3. Already completed child results remain attached to the swarm.
4. Running and queued children continue under the same scheduler.
5. The parent join is released, and the swarm registers as a
   `delegate-swarm` background task.

### Collaboration Controls

The model-facing control surface should be product-oriented, not debug-like.
Names below are conceptual and can be finalized during implementation planning.

```text
ListDelegates
WaitDelegate
MessageDelegate
InterruptDelegate
```

Rules:

- Controls target agent IDs, swarm IDs, or display names resolved through the
  current session registry.
- `MessageDelegate` can append instructions to a background or idle subagent.
- `WaitDelegate` blocks until completion, update, timeout, or interruption.
- `InterruptDelegate` cancels the current child turn but keeps the agent record.
- Foreground delegated agents do not need explicit wait calls; the parent is
  already joined to them.

## 4. TUI Transcript UX

Delegation is rendered as transcript content, using the same visual language as
thinking blocks and tool cards. It must not open a separate page or panel.

### Single Delegate

Single delegates use a compact Kimi-style agent transcript. The card is a
summary-first live view, not a raw dump of every child event.

```text
● Gibbs Agent Running (Implement Task 1: PlanBox border fix) · 3 tools · 24s · 25.6k tok
  Press Ctrl+B to run in background
  • Used Read (crates/neo-tui/src/transcript/plan_box.rs)
  ✗ Used Grep (from_spans|pub struct Span|pub struct Line)
  • Used Glob (crates/neo-tui/src/primitive*)
  ◌ Let me start by reading the current file to understand its structure.
  └ I'll implement this TDD task. First, let me read the current file.
```

Header shape:

```text
<status-dot> <DisplayName> Agent <State> (<task title>) · <tool-count> tools · <elapsed> · <tokens> tok
```

Status markers:

| Marker | Meaning |
| --- | --- |
| `●` | running |
| `✓` | completed |
| `✗` | failed |
| `◌` | queued, suspended, or waiting |

Body rows:

| Row | Meaning |
| --- | --- |
| `Press Ctrl+B to run in background` | shown only for foreground running delegates |
| `• Used <Tool>` | successful child tool call |
| `✗ Used <Tool>` | failed child tool call |
| `◌ <text>` | latest child assistant/thinking text |
| `└ <text>` | prior or wrapped child text tail |

Rules:

- Keep the latest child model text visible while running.
- Keep tool rows compact; long command/path details should truncate with ellipsis.
- Completed cards preserve the final child summary tail.
- Failed cards preserve the failing tool/error tail.
- Foreground cards show the `Ctrl+B` detach hint only while detach is available.
- Background cards do not show the detach hint; they show `/tasks` as the
  inspection path when helpful.
- Tool counts, elapsed time, and token counts update live.
- The child transcript is still backed by structured events; this compact card is
  only the default rendering.

### DelegateSwarm

Delegate swarms use a Kimi-style wide progress card. They start in an
`Orchestrating...` phase while arguments, item labels, and child runs are being
prepared, then move to `Working...` once children are active.

Orchestrating:

```text
─ Agent Swarm ─ Audit and fix Neo tool schemas ─────────────────────────

001 File write and directory tools: Write, Edit, Glob, ...
002 Search tools: Grep, Find. Neo files: crates/neo-agent...
003 Shell and terminal tools: Bash, Terminal. Neo files: ...
004 State, planning, collaboration, background, skill, ...

🟡 Orchestrating...
```

Working:

```text
─ Agent Swarm ─ Audit and fix Neo tool schemas ─────────────────────────

001 [..........] File write and directory tools: Write, Edit...
002 [######....] Search tools: Grep, Find. Neo files: crates/neo-agent...
003 [###.......] Shell and terminal tools: Bash, Terminal...
004 [..........] State, planning, collaboration, background...

🟡 Working...  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

Expanded child detail:

```text
▸ DelegateSwarm  Audit and fix Neo tool schemas                     41%
  4 agents running, 2 done, 0 failed, 1 rate-limited

  ● Gibbs   File write and directory tools       running   01:22 · 11.2k tok
      • Used Read (crates/neo-agent-core/src/tools/write.rs)
      ◌ Checking permission paths before proposing edits.

  ✓ Zeno    Search tools                         done      00:49 · 8.1k tok
      └ Found duplicated schema wording in grep/find docs.

  ◌ Hokke   Shell and terminal tools             suspended rate-limit
  ● Laber   State, planning, collaboration       running   02:04 · 15.7k tok
```

Display requirements:

- Running, queued, suspended, completed, failed, cancelled, and rate-limited
  states are visible.
- Child thinking/tool/output snippets appear under each child row when expanded.
- Global transcript expansion behavior applies to delegate cards too.
- Progress remains meaningful while some agents are complete and others are not.
- The progress estimator uses completed child durations, running tool activity,
  elapsed time, and capped unfinished progress rather than a naive count-only bar.
- Foreground blocking still keeps the TUI alive and animated.
- The compact swarm header should fit in one transcript card and degrade
  gracefully on narrow terminals.
- Item labels are deterministic from the swarm items; the LLM does not generate
  separate UI labels after tool call creation.
- Child detail expansion reuses the same global transcript expansion behavior as
  normal tool output and thinking blocks.

## 5. Background Task Browser Integration

Background delegates and swarms use Neo's existing `/tasks` surface. They do not
get a separate agent browser.

The shared task surface contains multiple task kinds:

```text
BackgroundTaskManager
  ├─ bash task
  ├─ ask-user task
  ├─ delegate task
  └─ delegate-swarm task
```

When a `Delegate` starts directly in background mode or is detached with
`Ctrl+B`, it registers a `delegate` background task. When a `DelegateSwarm`
starts directly in background mode or is detached with `Ctrl+B`, it registers a
`delegate-swarm` background task.

`/tasks` list rows should show:

```text
● delegate       Gibbs      running    Implement PlanBox border fix
● delegate-swarm swarm-02   running    Audit and fix Neo tool schemas
✓ delegate       Zeno       done       Review queue state
✗ delegate       Hokke      failed     Verify terminal tools
```

Selected delegate details:

```text
id:          agent_01kw...
kind:        delegate
name:        Gibbs
profile:     coder
mode:        background
status:      running
elapsed:     02:14
tokens:      25.6k
tools:       3
parent:      current session turn
task:        Implement PlanBox border fix
latest:      Reading transcript/plan_box.rs before editing.
```

Selected swarm details:

```text
id:          swarm_01kw...
kind:        delegate-swarm
status:      running
elapsed:     05:21
progress:    41%
children:    4 running, 2 done, 0 failed, 1 rate-limited
task:        Audit and fix Neo tool schemas
```

Preview output should reuse the same compact transcript rows as the inline card:
latest tool calls, latest child text, and terminal summary. The browser must not
load unbounded child logs into memory.

Task Browser actions:

- `S stop` maps to interrupt/cancel for running background delegates and swarms.
- `R refresh` updates from `BackgroundTaskManager`.
- `Enter/O output` opens the preview/output focus already defined by the Task
  Browser spec.
- Stopping a completed delegate or swarm is a no-op with a footer message.

Mailbox behavior:

- Completed background delegates and swarms remain visible in `/tasks`.
- Completion also creates a mailbox update for the parent agent.
- The parent should not automatically ingest large child transcripts.
- `WaitDelegate` or followup controls decide when a result enters model context.

This section extends the `/tasks` Task Browser design without changing its core
shape: `/tasks` remains a blocking human UI over the shared background task
manager.

## 6. Scheduler And Progress

The swarm scheduler should start conservatively and become more capable over
versions.

Initial rules:

- Default max concurrency: 4.
- Hard upper bound from config.
- Foreground swarm cancellation cancels unfinished foreground children.
- Background swarm cancellation cancels that swarm but not unrelated background
  agents.
- Results preserve item order.

Later rules:

- Rate-limit detection suspends affected children.
- Suspended children requeue with exponential backoff.
- The scheduler reduces effective concurrency under provider pressure.
- The progress estimator treats suspended/rate-limited agents as active but
  slower, not failed.

## 7. Agent Profiles And Permissions

Neo should use profile-based subagents rather than one untyped subagent mode.

Initial profiles:

| Profile | Purpose | Default tool stance |
| --- | --- | --- |
| `coder` | implement bounded code changes | normal Neo tool permissions |
| `explorer` | read/search/analyze only | no write/bash mutation by default |
| `planner` | produce design or implementation plan | no write/bash mutation by default |
| `reviewer` | inspect diffs, risks, tests | no write/bash mutation by default |
| `harness` | run scripted workflow steps | host-controlled capabilities |

Permission rules:

- Subagents inherit parent workspace, trust, and global denial boundaries.
- A subagent profile can narrow permissions but cannot widen beyond the parent.
- Git mutation is denied for subagents regardless of profile.
- Approval cards should identify the subagent display name and requested action.
- Background subagent approvals must not appear as anonymous main-agent actions.

## 8. Session Persistence

Multi-Agent state should persist in the session, not only in process memory.

Persist:

- agent identity
- display name
- parent/child path
- profile
- foreground/background mode
- lifecycle state
- mailbox entries
- final summary/result
- usage and timing metadata
- swarm item mapping
- compact transcript tail for task previews
- background task registration state

Do not persist provider wire events as the canonical API. Persist normalized Neo
events and derived state.

## 9. Workflow Harness Language

The workflow/harness layer should use **Lua** as Neo's one canonical embedded
workflow language.

### Why Lua

- Neo is a Rust local binary. Lua embeds cleanly through Rust crates such as
  `mlua` without requiring Node, Deno, or a separate runtime installation.
- Lua supports real control flow, loops, conditionals, functions, and coroutines.
- Lua is small enough to sandbox by exposing only Neo-provided host functions.
- Lua scripts can orchestrate agents without inventing a fragile YAML DSL.
- Lua avoids the large runtime, dependency, and permission surface of JS/TS.

### Why Not YAML

YAML is good for declaring static config, but workflow orchestration needs
branching, retries, loops, result inspection, and conditional followups. A YAML
workflow would quickly become a hidden programming language with worse tooling.

YAML can be used later for metadata around workflows, but not as the canonical
orchestration language.

### Why Not JS/TS

JS/TS matches Claude's public workflow direction and has broad familiarity, but
for Neo it adds the wrong first-order costs:

- heavier embedded runtime
- harder sandbox story
- dependency on Node/Deno if not embedded
- more complicated packaging for a local Rust binary
- larger API compatibility expectations

Neo can reconsider JS/TS only if workflows become a plugin ecosystem problem.
The built-in harness should start with Lua and stay canonical.

### Lua Host API Shape

Workflow scripts should call a small host API exposed by Neo:

```lua
local audit = neo.swarm({
  description = "Find risky runtime state transitions",
  role = "reviewer",
  mode = "foreground",
  items = {
    "runtime turn lifecycle",
    "queue and steer",
    "background task completion",
  },
  prompt_template = "Review {{item}} and return concrete risks."
})

if audit:has_failures() then
  neo.fail("Audit did not complete cleanly")
end

local fix = neo.delegate({
  role = "coder",
  mode = "foreground",
  task = "Fix the highest-confidence issue from the audit summary."
})

neo.verify("cargo run -p xtask -- test -p neo-agent-core runtime")
neo.report({ audit = audit:summary(), fix = fix:summary() })
```

Host API principles:

- Scripts can orchestrate Neo operations, not access arbitrary OS capabilities.
- File, shell, and tool access still go through Neo permissions.
- Workflow output is normalized into the transcript and session events.
- Workflow failures are resumable only at explicit step boundaries.
- Workflow resume does not restore an arbitrary Lua VM stack; it replays or
  continues from Neo-recorded workflow step state.
- Workflow APIs are versioned once, with no compatibility aliases for discarded
  experimental names.

## 10. Version Roadmap

### V1: Blocking Delegate And DelegateSwarm

Deliver:

- `MultiAgentRuntime`
- stable agent IDs
- deterministic display-name pool
- foreground `Delegate`
- foreground `DelegateSwarm`
- compact single-delegate transcript cards
- inline swarm transcript cards with orchestrating and working states
- aggregate result summaries
- subagent git-mutation denial
- focused fake-client tests

V1 deliberately blocks the parent by default and does not require long-lived
background residency.

### V2: Background Delegate

Deliver:

- `mode = background`
- `Ctrl+B` foreground-to-background detach for running delegates and swarms
- background agent registry
- `delegate` and `delegate-swarm` entries in `BackgroundTaskManager`
- list/wait/interrupt controls
- mailbox completion delivery
- `/tasks` rows, details, and previews for background delegates and swarms
- TUI state for background agents and swarms
- parent-safe result injection

V2 adds parallelism without changing V1 foreground behavior.

### V3: Followup Collaboration

Deliver:

- message/followup control
- subagent continuation after idle completion
- agent path display
- stable name retention across resumed sessions
- model guidance for when to wait versus continue

V3 makes background agents useful as collaborators, not just detached tasks.

### V4: Swarm Resume And Scheduler Maturity

Deliver:

- resume failed/partial swarm
- rate-limit suspension and retry
- adaptive concurrency
- full Bayesian-style progress estimator
- per-child usage and outcome report
- cancellation and partial-result semantics

V4 brings Kimi-level swarm reliability into Neo's runtime.

### V5: Lua Workflow Harness

Deliver:

- Lua workflow runner
- small sandboxed Neo host API
- workflow transcript cards
- workflow step persistence, not arbitrary VM checkpointing
- explicit verification/report helpers
- no YAML/JS compatibility layer

V5 turns repeated multi-agent procedures into reusable harnesses.

### Final Form

The complete system has:

- blocking and background delegation
- persistent agent tree
- deterministic named subagents
- inline transcript rendering
- swarm scheduler and progress estimator
- list/wait/message/interrupt controls
- profile-based permissions
- durable mailbox and session state
- Lua workflow harness
- no obsolete compatibility paths

## 11. Error Handling

Child failure:

- Foreground `Delegate` returns a failed tool result with child summary and error.
- Foreground `DelegateSwarm` returns aggregate partial results plus failed items.
- Background failure enters mailbox and task state.

Cancellation:

- Cancelling a foreground delegate cancels that child turn.
- Detaching a foreground delegate with `Ctrl+B` does not cancel it; it converts
  the same child run to background mode.
- Cancelling a foreground swarm cancels unfinished child turns and preserves
  completed summaries.
- Detaching a foreground swarm with `Ctrl+B` does not cancel it; it converts the
  same swarm run to background mode and preserves completed child results.
- Interrupting a background agent stops the current turn but keeps the agent.
- Stopping a background swarm from `/tasks` interrupts unfinished children and
  preserves completed child summaries.

Rate limits:

- V1 may report rate-limit failure directly.
- V4 adds suspension, retry, and adaptive concurrency.

Session resume:

- Resumed sessions restore agent names, terminal states, mailbox entries, and
  swarm summaries.
- Running process state that cannot be restored is marked interrupted/lost with
  a clear transcript state.

## 12. Testing Strategy

V1 tests:

- display-name allocation is deterministic
- foreground `Delegate` blocks parent until child completion
- foreground `DelegateSwarm` aggregates ordered results
- subagent git mutation is denied
- transcript events represent running and terminal child states
- single-delegate compact cards show state, tool count, elapsed time, tokens, and
  detach hint while running
- swarm cards show orchestrating and working states
- fake model clients can drive child completion deterministically

V2 tests:

- background delegate returns immediately
- `Ctrl+B` detaches a running foreground delegate without restarting it
- `Ctrl+B` detaches a running foreground swarm without restarting children
- detached delegates appear in `BackgroundTaskManager`
- detached and directly background swarms appear in `BackgroundTaskManager`
- `/tasks` renders delegate and delegate-swarm rows/details/previews
- wait blocks until update/completion/timeout
- interrupt stops active child turn
- mailbox completion is delivered once

V4 tests:

- partial swarm resume preserves completed items
- rate-limit suspension requeues without losing results
- progress estimate advances without claiming impossible completion

V5 tests:

- Lua workflows can call delegate/swarm/report helpers
- sandbox denies unexposed OS access
- workflow failures persist enough state for clear reporting
- verification helpers go through Neo's normal permission path

Use focused `cargo run -p xtask -- test -p <crate> <filter>` targets during
implementation. Do not widen to broad workspace gates unless the touched surface
requires it.

## 13. Open Implementation Decisions

These should be finalized in the implementation plan, not by changing the product
model:

- Exact Rust module placement after the recent runtime/TUI refactors.
- Exact tool schema field names.
- Whether agent paths use display names or stable short IDs in storage.
- Initial hardcoded name pool size.
- V1 concurrency default and config key name.
- Whether workflow support belongs in `neo-agent-core` or a small dedicated
  crate once Lua is introduced.

None of these decisions should introduce compatibility aliases for abandoned
experimental names.
